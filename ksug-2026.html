2부 — 사례 탐색 · 각 5~7분

"데이터가 보이기 시작하면 달라지는 것들"

"고칠 수 있는 건, 보이는 것뿐이다"

이 중 핵심 사례 2~3개는 데모로 직접 보여준다. 발표의 중심은 끝까지 사례다.

사례 ① 슬로우 쿼리 — 어딘지 몰랐던 병목

"서비스 전체가 느리다는 민원 — 막연하게 코드 훑기 반복"

문제 상황
  • 어디서 시간이 나가는지 감이 없음. 로그에는 "처리 완료"만 있음
  • N+1 문제만 있는 게 아니다 — 과도한 조인, 잘못된 인덱스도 마찬가지로 안 보인다

처음엔 인덱스가 잘 걸려 있었다. 그런데 기능이 추가될수록 테이블에 컬럼이 100개가 되고, 쿼리 패턴이 달라지면서 인덱스가 더 이상 타지 않게 됐다. 코드는 바뀌지 않았는데 쿼리는 점점 느려졌다. 처음부터 느린 게 아니었다 — 서비스가 성장하면서 느려진 것들이 있다.

GET /api/v1/orders (3,200ms)
├── findOrder           (5ms)    OK
├── SELECT order        (3ms)    OK
├── SELECT item         (2ms)    OK  ← × 1,000번 반복!
├── SELECT item         (2ms)    OK
...
└── (총 SQL 1,001회)              N+1 문제 / 인덱스 미적용 / 과도한 조인
⬆ 쿼리 개선 후 — p95 응답시간 3초 → 300ms

코드 리뷰에서는 안 보였다. 트레이스에서 처음 보였다.
로직은 바뀐 게 없는데 느려졌다면 — 데이터 양이 바꾼 것이다.

사례 ② 불필요한 쿼리 — 레거시 코드가 숨긴 것들

"처음부터 개발했으면 이렇게 안 만들었을 텐데"

레거시 코드가 만들어내는 패턴

처음부터 개발했다면 중복 쿼리를 보낼 이유가 없다. 하지만 레거시 코드를 다루다 보면 서비스 레이어가 복잡하게 분리돼 있고, 타고 타고 들어가다 보면 같은 데이터를 각자 따로 조회하고 있다는 걸 발견하게 된다.

서비스 레이어를 추상적으로 잘 나눠놨다고 생각하지만, 각 서비스가 독립적으로 마스터 데이터를 조회하고 있는 경우가 실무에서는 흔하다. 코드를 전부 타고 들어가 보지 않으면 절대 모른다.

// OrderService → ProductService → CategoryService
// 각각 독립적으로 보이지만...
SELECT * FROM category WHERE id = 1    ← OrderService에서 조회
SELECT * FROM category WHERE id = 1    ← ProductService도 조회
SELECT * FROM category WHERE id = 1    ← CategoryService도 조회
// 3번의 서비스 콜, 3번의 DB 쿼리 — 다 같은 데이터
SQL 통계에서 발견

실행 횟수 내림차순으로 정렬 → 같은 마스터 쿼리가 상위에 등장. 캐싱 추가 → DB 부하 40% 감소. 코드를 다 읽지 않고 데이터로 먼저 발견했다.

코드를 다 알지 못해도, 쿼리 통계는 안다.
레거시를 인수받은 첫 주에 이걸 보여주면 설득력이 생긴다.

사례 ③ 버그 찾기 — 재현이 안 됐던 에러

"간헐적 500 에러 제보 — 로컬에서 재현 불가"

문제 상황
  • 간헐적으로 500 에러 발생한다는 제보
  • 로컬에서 재현 불가
  • 에러 로그에 스택트레이스만 있고, 어떤 요청인지 알 수 없음

요청에 userId, itemId, itemStatus를 로그에 포함시키기 시작하면서 패턴이 보였다. 특정 유저가 특정 아이템을 특정 상태에서 조회할 때만 발생. 원인은 아이템 메타 정보의 버전 불일치였다.

// Before: 스택트레이스만
ERROR: NullPointerException at OrderService.java:142

// After: 컨텍스트 포함 (MDC 활용)
ERROR: 주문 처리 실패
  userId="user-7892"
  itemId="item-3301"
  itemStatus="MIGRATED_V1"
  path="/api/v1/orders"
  error="NullPointerException"
  ← 구버전 메타 포맷 아이템 + 신규 로직 조합에서만 발생

grep만으로는 로그의 패턴을 좁혀나가기 힘들다. key로 좁히고, 시간대로 비교하고, 통계를 내는 식의 분석이 필요하다. 구조화된 로깅이 있어야 그게 가능해진다.

⬆ 원인 특정까지 걸린 시간 며칠 → 1시간

로그에 컨텍스트가 있어야 분리하고, 좁혀나갈 수 있다.
grep은 "찾기"는 되지만 "분석"은 안 된다.

사례 ④ 배포 전후 비교 — 개선됐는지 어떻게 아나

"리팩토링 완료 → '아마 빨라졌겠지' → PR 머지"

배포 전후 응답시간, 에러율, DB 호출 수를 나란히 보면 개선됐는지 확인할 수 있고, 예상 못한 regression이 배포 직후 잡힌다.

비포-애프터를 수치로 비교 못 하면 생기는 일
  • 로그파일에 다들 그냥 넘겨도 되는 에러 메시지들 있지 않은가. "이렇게 느린데 문제 있는 API 아니야?" 싶은 것들도 있다
  • 자기 회사에서만 "정상"이라고 합의된 것들. 그런 것들에 대한 비교 기준이 필요하다
  • "느낌"으로 — "앗 배포하고 뭐가 달라졌어" 라고 하는 건 힘들다. 숫자가 있어야 한다
// 배포 전
p50: 180ms  p95: 3,200ms  DB calls/req: 8.3

// 배포 후 (v2.1.0)
p50: 160ms  p95: 290ms   DB calls/req: 1.1

// 예상 못한 regression
[DEPLOY v2.1.1] ─────────────────────────────────
p50: 170ms  p95: 8,400ms  ← 신규 기능 배포 후 급등

배포가 "끝"이 아니라 "확인의 시작"이 됐다.
이상함을 "느끼는 것"이 아니라 "발견하는 것"으로 바뀐다.

사례 ⑤ 응답 속도 분포 — 평균은 거짓말한다

"평균 응답시간 200ms → '빠른 서비스'라고 생각했다"

구간응답시간의미
평균 (mean) 200ms 거짓말하는 숫자
P50 95ms 절반의 사용자
P95 2,100ms 100명 중 5명이 2초 이상 대기
P99 8,400ms 100명 중 1명이 8초 이상 대기
평균 데이터로 보고하면 생기는 일
  • 사장님 PC에서만 느린 경우 — 평균은 괜찮다고 했는데 큰일을 치를 수 있다
  • 고객 관점에서 보면, 헤비 유저가 돈이 되는 경우가 많다. 다른 고객은 괜찮지만 헤비 유저가 느리면 매출에 직격
  • 이런 경우 영업이 고객의 컴플레인을 가져온다. 먼저 알 수 있으면 좋겠죠

분포에서 "긴 꼬리" 구간 → 해당 구간 요청의 공통 패턴 드릴다운 → 특정 조건에서만 발생하는 것을 발견.

평균은 거짓말한다. 분포가 진실을 말한다.
보는 방법이 달라지면, 보이는 게 달라진다. 영업이 가져오기 전에 먼저 발견하자.

사례 ⑥ 커넥션 풀 고갈 — 쿼리는 1ms인데 응답이 25초인 이유

"SQL 최적화를 하러 들어갔다가 전혀 다른 걸 발견했다"

증상

현상: 특정 API의 응답시간이 들쑥날쑥. 평소엔 50ms, 가끔 20~30초 이상.

이상한 점: 느릴 때 트레이스를 열어보면 — SQL 수행 자체는 1~6ms. 그런데 경과 시간은 25,000ms.

트레이스 구간소요 시간의미
DB Connection 획득 24,931ms 커넥션 대기
SQL 실행 1~6ms 쿼리 자체는 정상
전체 경과 25,057ms 거의 전부가 대기
발견 경로

원인: DB Pool 최대치(20개)에 Active 커넥션이 상한선까지 점유 중. 신규 요청은 빈 풀이 생길 때까지 대기.

메트릭스 차트: DB Pool 개수 vs DB Active 개수를 나란히 보면 — Active가 Pool 한도에 붙어서 눌려있는 패턴이 보인다. Connection Leak이라면 Active가 우상향으로 계속 증가한다.

레거시에서 종종 보이는 패턴이 하나 더 있다. 코드를 DB화하는 경우 — 코드성 데이터(상태값, 코드 목록)를 매번 DB에서 조회하는 것이다. DB는 비싼 자원인데, 가벼운 코드 조회나 상태 조회 때문에 전체 시스템 퍼포먼스가 내려갈 수 있다.

커넥션 풀 같은 경우도 처음엔 문제가 안 된다. 하지만 서비스의 성공과 함께 사용량이 계속 늘어나면 문제가 된다. 로직이 변경되지 않아도 문제가 발생할 수 있다.

쿼리 튜닝만이 답이 아니다.
코드 바깥 — 인프라 설정 문제를 APM에서 발견했다. 서비스가 성장하면서 코드 변경 없이도 터질 수 있다.

사례 ⑦ 히트맵 패턴 — 그래프 모양이 장애 종류를 말해준다

"히트맵은 느린 요청을 찾는 도구가 아니다. 모양 자체가 진단이다"

패턴모양의미
세로줄 │ 특정 시각에 집중 Lock 해소 → 대기 트랜잭션이 한꺼번에 종료
가로줄 ── 특정 응답시간에 집중 타임아웃 설정값이 그대로 드러남 (예: 10초)
폭주 패턴 ▲ 전체적 상승 과부하 → 응답시간 전체 증가
플라잉 패턴 〜 파도처럼 반복 공유 자원(로그, 파일 등) 간헐적 부족
Lock 진단 팁

세로줄 패턴 분기점: 단일 인스턴스에서만 발생하면 AP 내부 Lock. 모든 인스턴스에서 동시에 발생하면 외부 DB Lock. 이걸 구분하는 것만으로 조사 범위가 반으로 줄어든다.

데이터를 보는 방법이 달라지면, 질문하는 방식이 달라진다.
"왜 느리지?"가 아니라 "이건 세로줄인가, 가로줄인가"로 시작하게 된다.

사례 ⑧ CPU 스파이크 + 스레드 덤프 — 100%의 세 가지 원인

"CPU가 갑자기 100%가 됐다 — 어디서 시작하지?"

CPU 급증의 원인은 세 가지로 구분된다
원인 유형확인 위치신호
① 힙 메모리 가득 참 대시보드 힙 메모리 그래프 GC가 반복 실행 → CPU 급증
② 실제 요청이 많음 TPS 지표 요청 수와 CPU 사용량이 비례
③ 코드 내 비효율 스레드 목록/덤프 무한 루프, 비효율 알고리즘, 과도한 직렬화
스레드 덤프 분석 흐름

스레드 CPU 시간 확인 → 새로고침 → 증가량 내림차순 정렬 → CPU 많이 쓰는 스레드 특정 → 스냅샷에서 자주 호출되는 메소드 확인

실무 사례 — 역직렬화 비용

문자열이 포함된 pack 데이터의 직렬화/역직렬화 시 문자열 할당이 발생한다. 대부분의 경우엔 문제없지만, 특정 경우에 문자열이 매우 클 때 역직렬화 비용이 급격히 커져서 CPU 스파이크를 유발했다. 결국 필요 없는 필드는 역직렬화를 건너뛰게 해서 해결했다.

이런 문제는 메모리 분석과 스택 트레이스 분석이 있어야만 발견할 수 있다.

히트맵이 응답시간 최적화의 핵심이라면
스레드 덤프는 CPU 최적화의 핵심이다.

사례 ⑨ 외부 HTTP Call 지연 — 내 코드는 정상인데 왜 느린가

"트레이스를 보니 — 지연 구간 전체가 외부 API였다"

증상

응답시간이 간헐적으로 느림. 내부 SQL은 정상. 트레이스에서 HTTP Call 요약 탭을 열어보니 외부 결제 API 호출 1건이 합계 시간의 90%를 차지하고 있었다.

외부 vs 내부 지연 판별

호출 건수는 적은데 합계 시간이 크다 → "외부 서비스가 간헐적으로 느려지고 있다"는 신호. 이걸 발견하면 두 가지 결론이 나온다: 우리 코드 문제가 아니다는 증명, 또는 서킷 브레이커 도입의 근거.

"도둑잡기"에서 빠르게 벗어나기

외부 API 연동 시 이슈는 항상 많다. 중요한 건 우리 잘못이 아니라는 걸 입증하는 것이다. 일할 때도 백엔드-프론트엔드 간에 "내 잘못인지 아닌지" 얘기하는 게 중요하지 않은가. 도둑잡기에서 빠르게 벗어나는 것이 핵심이다. APM 트레이스 한 장이면 된다.

에이전트 하나 붙였더니
"우리 잘못이 아니었다"를 증명했다.

사례 ⑩ 에러 최초 생성 시간 — 이 에러, 언제부터 있었던 걸까

"어제 배포 이후 에러가 생겼다고 했는데 — 통계를 보니 2주 전부터 있었다"

상황

장애 슬랙 알람 → "어제 배포가 문제다" 주장 → 통계 > 에러 메뉴에서 최초 생성 시간 컬럼 확인 → 해당 에러 클래스의 최초 발생: 배포 14일 전.

에러 통계 구조

에러 클래스 + 트랜잭션 + 메시지를 키로 묶어서 — 조회 기간 중 해당 조합이 언제 처음 발생했는지를 보여준다. 5분마다 최대 1,000가지 에러를 통계화.

에러인데 에러가 아닌 것들

문제가 없을 때는 그냥 넘어가는 것들도, 장애 상황이 되면 "이게 원인 아니냐"는 추궁을 받기 일쑤다. 예를 들어 배포 후 애플리케이션 캐시 초기화로 캐시 미스가 나는 데이터를 에러로 로깅하고 있었다면 — 평소에는 그냥 넘어가던 것들이, 배포 장애가 난 날에는 의미 있는 장애 원인처럼 보이기 쉽다.

기간별 통계를 통해 에러지만 에러가 아닌 것들, 문제가 아니었던 것들을 명확히 설명할 수 있으면 — 불필요한 디버깅 시간을 줄일 수 있다.

에러는 발견된 날이 발생한 날이 아닐 수 있다.
"누가 만든 버그야" 싸움의 종결자.

사례 ⑪ IP 트랜잭션 통계 — 슬로우 쿼리가 없는데 DB가 왜 무겁지

"특정 내부 배치 서버가 /api/products를 분당 수천 번 호출하고 있었다"

증상

슬로우 쿼리 없음, 개별 트랜잭션 정상, 그런데 DB 부하가 높음. 통계 > IP 트랜잭션에서 호출 건수 내림차순 정렬 → 내부 IP 한 개가 전체 요청의 60%를 차지하고 있었다.

IP 트랜잭션 통계가 보여주는 것

IP별 호출 건수 + 에러 수 + 평균 응답시간. "사람이 아니라 기계가 API를 폭격하고 있었다"는 패턴. 보안 위협일 수도 있고, 배치 잡의 루프 버그일 수도 있다.

기능은 정상인데, 원인이 고객인 경우

기능적으로 정상이더라도, 우리 팀이 만들지 않은 — 또는 고객이 원인이 되는 경우도 많다. 예를 들어 고객이 우리 시스템을 포함해서 매우 헤비하게 부하 테스트를 돌리는 경우. "우리 버그인가, 뭐가 문제인가" 했을 때 특정 고객의 특정 비정상 사용 패턴으로 문제를 좁힐 수 있다.

운영 데이터가 이상 징후를 먼저 발견했다.
보안도, 고객 이슈도 — 결국 모니터링이다.

열한 사례의 공통점

코드를 아무리 잘 짜도, 운영 데이터 없이는 개선할 수 없는 것들이 있다.

슬로우 쿼리도, 커넥션 풀도, CPU 스파이크도, 외부 API 지연도 — 전부 운영 중에 발견된 것들이다.
데이터가 쌓이기 시작하면 — 개선이 개선을 부르는 사이클이 돌기 시작한다.

"필요하다는 걸 아는 것 자체가 시작이다"

솔직하게 말하겠습니다. 앞에서 보여드린 사례 대부분은 로그만으로는 불가능합니다.

로그만으로 할 수 있는 것의 한계
  • grep으로 에러 메시지를 찾을 수는 있다. 하지만 패턴 분석, 시간대별 통계, key 기반 필터링은 로그 수집 인프라(ELK, Loki, Sentry)가 있어야 한다
  • 슬로우 쿼리 로깅을 켤 수는 있다. 하지만 "이 API 안에서 SQL이 1,001번 실행됐다"는 트레이스 없이는 안 보인다
  • 커넥션 풀 대기시간 24초, 히트맵 패턴, 외부 API 지연 비중 — 이건 APM 없이는 볼 수 없다

그래서 이야기의 핵심은 "로그를 잘 찍으세요"가 아닙니다. "이런 게 보여야 한다는 걸 아는 것"이 시작입니다. 그걸 알아야 회사에 요구할 수 있고, 직접 깔아볼 수도 있습니다.

"지금 당장" 할 수 있는 최소 3가지

도구를 도입하기 전이라도, 코드에서 시작할 수 있는 것들이 있습니다. 이것들은 나중에 어떤 도구를 쓰든 그대로 활용됩니다.

  1. 로그에 컨텍스트 넣기 MDC에 userId, requestId 추가. 이게 있어야 나중에 어떤 로그 수집기를 쓰든 필터링과 분석이 가능해진다. 사례 ③의 핵심.
  2. Actuator + Micrometer 켜기 Spring Boot라면 의존성 두 줄이면 된다. 이게 있어야 Prometheus든 Datadog이든 메트릭을 수집할 수 있는 상태가 된다.
  3. 오픈소스로 한번 체험해보기 Grafana 스택이든 Pinpoint든 docker-compose로 로컬에서 먼저 한번 돌려보기. 어떤 상황이더라도 트레이스를 한 번 직접 보면, 왜 운영 데이터가 필요한지 감이 훨씬 빨리 온다.
