Level 0 → 1 8분

에피소드 1: "에러 500개, 어디서 터지는 건지?"

문제 상황
  • 신기능 배포 후, 특정 조건에서만 NPE 발생
  • 에러율이 슬금슬금 올라가고 있음
  • grep ERROR app.log → 500개. 어디서? 누구한테?
  • 로그에 userId도, path도 안 남김 — grep으로 특정 불가
  • 결국 CS 인입 후에야 알게 됨
grep ERROR app.log | wc -l
→ 500개. 그래서?

grep "NullPointerException" app.log | grep "userId"
→ 안 나온다. 로그에 userId를 안 남겼으니까.

"에러가 나고 있다는 건 알겠는데, 어떤 조건에서 나는지 모르면 고칠 수가 없습니다."

사실 저도 알고 있었는데 안 했습니다. "지금 돌아가니까 나중에 하지 뭐" 하고요. CS 인입 세 번 당하고 나서 바꿨습니다.

Structured Log로 바꾸면
  • path 별 에러 집계 → /api/v1/orders에서만 집중
  • userId 별 필터링 → 프로필에 주소가 null인 유저만 발생
  • 배포 전후 비교 → v2.3.1 배포 이후에만 발생
// build.gradle
implementation 'net.logstash.logback:logstash-logback-encoder:7.4'
// Before
log.error("주문 처리 실패: " + e.getMessage());

// After
import static net.logstash.logback.argument.StructuredArguments.*;

log.error("주문 처리 실패",
    kv("orderId", orderId),
    kv("userId", userId),
    kv("path", request.getPath()),
    kv("error", e.getClass().getSimpleName()));
// 출력 결과 (JSON)
{
  "level": "ERROR",
  "message": "주문 처리 실패",
  "orderId": "ord-12345",
  "userId": "user-789",
  "path": "/api/v1/orders",
  "error": "NullPointerException"
}

"로그 한 줄만 바꾸면, 500개에서 타겟 3줄로 좁힐 수 있습니다."

스크린샷: 로그 검색 도구에서 structured log 필터링 화면
Level 1 → 2 8분

에피소드 2: "우리 코드 문제 아닌 것 같은데요 — 근데 증거가 있나요?"

문제 상황 — 외부 의존성
  • 결제 API 응답이 간헐적으로 30초
  • 우리 코드는 멀쩡한데, 외부 PG사 API가 타임아웃
  • "우리 쪽 문제 아닌 것 같은데요"라고 말하려면 증거가 있어야 함
  • Structured Log로는 "결제 API가 느리다"까지만 알 수 있음. 근데 왜?

먼저 결과를 보여드리겠습니다. Tracing을 켜면 이렇게 보입니다:

POST /api/v1/payments (30,042ms)
├── validateOrder        (30ms)   OK
├── calculateShipping    (12ms)   OK
├── processPayment      (29,800ms) FAIL ← 여기!
│   └── HTTP Call: pg.example.com  (29,750ms) ← 외부 PG사 타임아웃
└── finalizeOrder        (200ms)  OK

"아, 우리 코드가 아니라 PG사 API가 느린 거구나."
이게 grep으로는 절대 안 나오는 정보입니다. 그리고 이게 증거입니다.

N+1도 같은 방식으로 잡힙니다. 에러도 안 나고, 로그에도 안 남는 그 느림의 원인이 숫자로 나옵니다:

GET /api/v1/orders (3,200ms)
├── findOrder           (5ms)    OK
├── SELECT order         (3ms)    OK
├── SELECT item          (2ms)    OK ← × 1,000번 반복!
...
└── (총 SQL 1,001회)              N+1 PROBLEM
Tracing 도입 방법
  • OpenTelemetry + Grafana Tempo 조합으로 오픈소스로도 구축 가능
  • 요청이 끝난 뒤 전체 흐름을 추적하고 분석할 수 있음
  • 원인 파악, 병목 위치, 외부 의존성 구분 — 여기까지는 됩니다

근데 오픈소스 트레이싱이 못 하는 게 하나 있습니다.
"지금 이 순간" 뭐가 막혀있는지.
요청이 끝나야 데이터가 쌓입니다. 장애는 지금 일어나고 있는데.

스크린샷: 트레이스 뷰 — 외부 PG사 타임아웃 (부모/자식 스팬)
스크린샷: 트레이스 뷰 — N+1 쿼리 (SQL 반복 호출)

실무 사례 — 트레이싱 없이는 못 잡는 것들

발표에서는 슬라이드 한 장씩 빠르게 소개. "이런 경우 있으셨나요?" 질문과 함께.

사례 1 재고 서비스가 특정 시간대만 느리다

주문 API는 멀쩡한데 오후 2시마다 P95가 튄다. 로그엔 "재고 조회 완료"만 있음.

POST /api/v1/orders (4,200ms)  ← 평소엔 300ms
├── validateUser        (20ms)   OK
├── checkInventory     (3,900ms) ← 여기!
│   └── HTTP Call: inventory-service (3,850ms) ← 배치 작업 중
└── createOrder         (280ms)  OK

재고 서비스가 오후 2시에 배치 집계 작업을 돌리고 있었음. 로그만 봤으면 "가끔 느린 거겠지"로 넘어갔을 것.

사례 2 회원가입은 됐는데 인증 메일이 안 온다

API 응답은 200 정상. 에러 없음. 근데 사용자 CS: "메일이 안 와요."

POST /api/v1/signup (120ms)  200 OK  ← 코드상 문제없음
├── createUser          (80ms)   OK
└── publishEvent → email-queue  ← 여기서 끊김

// 비동기 큐 이후는 요청-응답 밖
email-queue → email-worker  ← 워커 다운, 큐 적체 8,000건

비동기 처리라 요청은 성공. 트레이싱 없으면 큐 이후가 블랙박스.

사례 3 각각은 빠른데 합치면 느리다

상품 상세 API가 3초. 각 서비스 로그엔 다 "1초 이내 완료".

GET /api/v1/product/123 (2,800ms)
├── getProduct          (50ms)   OK
├── getReviews          (900ms)  OK  ← 순차 실행
├── getRecommendations  (950ms)  OK  ← 순차 실행
└── getInventory        (880ms)  OK  ← 순차 실행

에러 없음. 각 서비스 정상. 근데 직렬 호출이라 합산 3초. 병렬로 바꾸면 950ms. 트레이싱 없이는 원인이 코드에 있는지 외부에 있는지도 모름.

사례 4 DB 커넥션 풀이 꽉 찼다

에러율 0%. 전체적으로 느려짐. 로그엔 아무것도 없음.

// Active Transaction 목록 (지금 이 순간)
30개 요청이 전부 WAITING_FOR_DB_CONNECTION 상태

POST /api/v1/orders      (8,200ms)  WAITING
POST /api/v1/orders      (7,900ms)  WAITING
POST /api/v1/orders      (7,650ms)  WAITING
... (27개 더)

커넥션 풀 사이즈 10, 동시 요청 40. 로그 기반 트레이싱은 요청이 끝나야 보임. APM 실시간 Active Transaction이 있어야 "지금 이 순간"을 잡을 수 있음.

공통점

에러가 없다. 로그가 깨끗하다. 근데 느리다.
보이지 않으니까 고칠 수가 없다.

Level 2 → 3 6분

에피소드 3: "에러는 없는데 — 왜 결제가 안 되는 거죠?"

문제 상황
  • 배포 후 에러율 정상. 로그도 깨끗함
  • 근데 CS팀: "결제 완료가 안 된다는 문의가 계속 들어와요"
  • 내 자리에서는 결제 잘 됨. 재현이 안 됨
  • 대부분의 사용자는 괜찮은데 — 어떤 사용자가, 어떤 조건에서?
Dashboard가 있으면
  • P50 = 200ms, P95 = 10초 → 대부분은 빠른데, 상위 5%가 문제
  • 엔드포인트별 비교 → /api/payments만 P95 급등
  • 배포 전후 추이 → v2.3.1 이후부터 시작됨

P50 / P95 / P99란?
100명이 API를 호출했을 때, P50은 50번째로 느린 사람의 응답시간, P95는 95번째의 응답시간입니다. P50이 빨라도 P95가 느리면 — 100명 중 5명은 10초를 기다리고 있다는 뜻입니다.

"에러율은 정상입니다. 하지만 /api/payments의 P95가 10초입니다. 100명 중 5명이 결제 페이지에서 10초를 기다리다 이탈하고 있습니다. 원인은 외부 PG사 타임아웃이고, 대응 중입니다."

이 한마디를 할 수 있느냐 없느냐.
그 차이가 대시보드 하나입니다.

스크린샷: 응답 시간 히스토그램 — P50 / P95 / P99 비교

에피소드 1~3에서 한 것 — 보이는 범위를 한 단계씩 넓힌 것.

Level볼 수 있게 된 것필요한 것
0 → 1어디서, 누구한테Structured Log
1 → 2왜, 어디서 막히는지Tracing
2 → 3분포와 추이Dashboard

이걸 한 단어로 부르면 Observability입니다.

도구 이름이 아니라, "보이는 범위를 넓히는 사고방식"의 이름입니다.

Level 4 — Real-time 12분

실무에서 바로 쓰기

A. WhaTap으로 보는 실시간 모니터링 (8분)

솔직히 말씀드리면, 저는 WhaTap이라는 모니터링 서비스를 만드는 회사에서 일하고 있어서 편향이 있을 수 있습니다. 그 점 감안해서 들어주세요.

다만 아까 이야기한 그 한 가지 — "지금 이 순간 뭐가 막혀있는지" — 그게 실제로 어떻게 보이는지는 보여드리고 싶었습니다.

대시보드 — 한눈에 보이는 서비스 상태

에이전트 설치만 하면 대시보드가 자동 구성. 앞서 이야기한 "P95가 10초" 같은 답을 바로 할 수 있는 화면.

스크린샷: WhaTap APM 대시보드 — Speed Meter, 에러율, 응답시간

히트맵 + 액티브 트랜잭션 — 지금 이 순간 뭐가 막혀있는지

오픈소스 트레이싱은 요청이 끝난 뒤에 데이터를 봅니다. 장애 상황에서는 "지금 이 순간 뭐가 막혀있는지"를 봐야 합니다.

각 트랜잭션이 SQL 실행 중인지, 외부 API 대기 중인지, 코드 실행 중인지 즉시 분류됩니다.

스크린샷: WhaTap 히트맵 + Active Transaction 목록

"장애가 터진 그 순간, '지금 30개 요청이 DB 커넥션 대기 중이다'가 보이면 커넥션 풀 설정 문제라는 걸 바로 알 수 있습니다."

SQL 분석 — JPA 뒤에서 뭐가 일어나는지

"JPA 쓰시는 분, Hibernate가 뒤에서 어떤 쿼리를 만드는지 본 적 있으세요?"

실행된 쿼리, 실행 시간, 실행 빈도를 자동 수집. N+1 문제를 숫자로 바로 확인 가능.

스크린샷: WhaTap SQL 통계 — 슬로우 쿼리 Top 10

docker-compose 하나면 이 전체가 올라갑니다. 무료입니다.

C. 다음 단계 (2분)

Structured Log가 잘 남아있으면, AI Agent가 활용할 수 있는 기반이 됩니다. Honeycomb MCP, Datadog MCP 등 이미 프로덕션에서 사용 중. 반대로 log.error("결제 실패: " + e.getMessage()) 뿐이면 AI도 할 수 있는 게 없습니다.

"지금 로그 한 줄을 바꾸는 게, 미래에 AI Agent를 활용할 수 있는 기반이 됩니다."

Level 5 — Agentic 8분

다음 질문: 사람이 안 봐도 된다면?

지금까지 이야기한 것들의 전제는 하나였습니다. 사람이 본다.

대시보드를 열고, 히트맵을 읽고, 판단하고, 대응하는 사람.

그 전제가 바뀌고 있습니다.
에이전트가 직접 모니터링 데이터를 소비하고, 판단하고, 대응합니다.

지금 일어나고 있는 일

Agentic Monitoring 패러다임
  • Honeycomb MCP, Datadog MCP — 에이전트가 직접 트레이스/메트릭 조회
  • "P95가 튀고 있다" → 에이전트가 원인 분석 → 자동 대응 또는 PR 생성
  • 사람은 대시보드를 보는 게 아니라 에이전트의 판단 결과를 승인
// 지금 패러다임
P95 급등 감지 → 대시보드 알림 → 개발자 확인 → 원인 분석 → 대응
                                    ↑ 사람이 개입하는 지점

// 다음 패러다임
P95 급등 감지 → 에이전트: 트레이스 조회 → 원인 특정 → 해결 방안 제시 → 개발자 승인

그러면 Observability는 의미 없어지는 건가요?

반대입니다. 더 중요해집니다.
에이전트가 보려면 — 볼 수 있는 데이터가 먼저 있어야 합니다.

에이전트는 없는 데이터를 만들어낼 수 없습니다. log.error("결제 실패: " + e.getMessage()) 뿐인 로그에서 에이전트도 할 수 있는 게 없습니다.

Observability의 정의가 바뀌는 겁니다.

패러다임Observability의 목적설계 기준
지금사람이 읽을 수 있게시각화, 대시보드
다음에이전트가 판단할 수 있게구조화, 컨텍스트 밀도

WhaTap이 가고 있는 방향

대시보드를 없애는 게 아닙니다. 대시보드 너머를 설계하는 겁니다.

에이전트가 소비하는 모니터링 데이터

사람이 읽는 시각화 → 에이전트가 쿼리할 수 있는 구조화된 데이터로. 트레이스, 메트릭, 로그가 MCP를 통해 에이전트의 감각기관이 됩니다.

자율 대응 워크플로우

감지 → 분석 → 대응의 루프를 에이전트가 수행. 사람은 임계값을 정의하고 결과를 승인합니다. 장애 대응 시간이 분 단위에서 초 단위로.

"지금 로그 한 줄을 구조화하는 게,
