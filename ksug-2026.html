Level 0 → 1 8분

에피소드 1: "에러 500개, 어디서 터지는 건지?"

문제 상황
  • 신기능 배포 후, 특정 조건에서만 NPE 발생
  • 에러율이 슬금슬금 올라가고 있음
  • grep ERROR app.log → 500개. 어디서? 누구한테?
  • 로그에 userId도, path도 안 남김 — grep으로 특정 불가
  • 결국 CS 인입 후에야 알게 됨
grep ERROR app.log | wc -l
→ 500개. 그래서?

grep "NullPointerException" app.log | grep "userId"
→ 안 나온다. 로그에 userId를 안 남겼으니까.

"에러가 나고 있다는 건 알겠는데, 어떤 조건에서 나는지 모르면 고칠 수가 없습니다."

Structured Log로 바꾸면
  • path 별 에러 집계 → /api/v1/orders에서만 집중
  • userId 별 필터링 → 프로필에 주소가 null인 유저만 발생
  • 배포 전후 비교 → v2.3.1 배포 이후에만 발생
// Before
log.error("주문 처리 실패: " + e.getMessage());

// After
log.error("주문 처리 실패: orderId={}, userId={}, path={}, error={}",
    orderId, userId, request.getPath(), e.getClass().getSimpleName());
// 출력 결과
{
  "level": "ERROR",
  "orderId": "ord-12345",
  "userId": "user-789",
  "path": "/api/v1/orders",
  "error": "NullPointerException"
}

"로그 한 줄만 바꾸면, 500개에서 타겟 3줄로 좁힐 수 있습니다."

스크린샷: 로그 검색 도구에서 structured log 필터링 화면
Level 1 → 2 8분

에피소드 2: "느려지고 있는데, 왜 느린지 모른다"

문제 상황 A — N+1
  • 신기능에 @OneToMany 관계 추가. 개발 환경에선 데이터 10건이라 문제 없음
  • 운영에서 데이터 1,000건 → 쿼리 1,001번 실행
  • 해당 API만 점진적으로 느려짐. 에러도 안 나고, 로그에도 안 남음
문제 상황 B — 외부 의존성
  • 결제 API 응답이 간헐적으로 30초
  • 우리 코드는 멀쩡한데, 외부 PG사 API가 타임아웃
  • "우리 쪽 문제 아닌 것 같은데요"라고 말하려면 증거가 있어야 함

Structured Log로 "결제 API가 느리다"까진 알았다. 근데 왜?

Tracing을 켜면
  • 요청의 전체 흐름이 부모/자식 스팬으로 보임
  • 어디서 시간이 소비되는지 숫자로 확인 가능
  • 내부 문제인지, 외부 문제인지 즉시 구분
POST /api/v1/payments (30,042ms) ├── validateOrder (30ms) OK ├── calculateShipping (12ms) OK ├── processPayment (29,800ms) FAIL ← 여기! │ └── HTTP Call: pg.example.com (29,750ms) ← 외부 PG사 타임아웃 └── finalizeOrder (200ms) OK

"아, 우리 코드가 아니라 PG사 API가 느린 거구나."
이게 grep으로는 절대 안 나오는 정보입니다.

N+1도 마찬가지:

GET /api/v1/orders (3,200ms) ├── findOrder (5ms) OK ├── SELECT order (3ms) OK ├── SELECT item (2ms) OK ← × 1,000번 반복! ... └── (총 SQL 1,001회) N+1 PROBLEM

Spring Boot에서는 이런 코드면 됩니다:

@Observed(name = "order.process", contextualName = "processOrder")
public Order processOrder(String id) {
    processingSteps.validateOrder(order);
    processingSteps.calculateShipping(order);
    processingSteps.finalizeOrder(order);
}

"Spring Boot 4는 @Observed 어노테이션 하나로 자동 추적.
Micrometer + OpenTelemetry 기본 내장, 설정 최소."

스크린샷: 트레이스 뷰 — 외부 PG사 타임아웃 (부모/자식 스팬)
스크린샷: 트레이스 뷰 — N+1 쿼리 (SQL 반복 호출)
Level 2 → 3 6분

에피소드 3: 사장님 — "느리다"

문제 상황
  • 사장님 전화: "화면이 느려요"
  • 내 자리에서는 200ms. 재현이 안 됨
  • "제 자리에서는 괜찮은데요..." 하고 넘기거나, 데이터로 답하거나
Dashboard가 있으면
  • P50 = 200ms, P95 = 10초 → "대부분은 괜찮은데 상위 5%가 문제"
  • 엔드포인트별 비교 → /api/payments만 느림
  • 배포 전후 추이 → v2.3.1 이후 P95 급등

"사장님, 90%는 정상입니다. /api/payments의 P95가 10초인데, 외부 PG사 타임아웃이 원인입니다. 대응 중입니다."

이 한마디를 할 수 있느냐 없느냐.
그 차이가 대시보드 하나입니다.

스크린샷: 응답 시간 히스토그램 — P50 / P95 / P99 비교

에피소드 1~3에서 한 것 — 보이는 범위를 한 단계씩 넓힌 것.

Level볼 수 있게 된 것필요한 것
0 → 1어디서, 누구한테Structured Log
1 → 2왜, 어디서 막히는지Tracing
2 → 3분포와 추이Dashboard

이걸 한 단어로 부르면 Observability입니다.

도구 이름이 아니라, "보이는 범위를 넓히는 사고방식"의 이름입니다.

Level 4 — Real-time 12분

실무에서 바로 쓰기

A. WhaTap으로 보는 실시간 모니터링 (8분)

"지금까지 '보이게 만드는 과정'을 이야기했습니다. 저는 WhaTap이라는 모니터링 서비스를 만드는 회사에서 일하고 있는데요, 실무에서 이것들이 어떻게 보이는지 보여드리겠습니다."

대시보드 — 한눈에 보이는 서비스 상태

에이전트 설치만 하면 대시보드가 자동 구성. 앞서 이야기한 "P95가 10초" 같은 답을 바로 할 수 있는 화면.

스크린샷: WhaTap APM 대시보드 — Speed Meter, 에러율, 응답시간

히트맵 + 액티브 트랜잭션 — 지금 이 순간 뭐가 막혀있는지

오픈소스 트레이싱은 요청이 끝난 뒤에 데이터를 봅니다. 장애 상황에서는 "지금 이 순간 뭐가 막혀있는지"를 봐야 합니다.

각 트랜잭션이 SQL 실행 중인지, 외부 API 대기 중인지, 코드 실행 중인지 즉시 분류됩니다.

스크린샷: WhaTap 히트맵 + Active Transaction 목록

"장애가 터진 그 순간, '지금 30개 요청이 DB 커넥션 대기 중이다'가 보이면 커넥션 풀 설정 문제라는 걸 바로 알 수 있습니다."

SQL 분석 — JPA 뒤에서 뭐가 일어나는지

"JPA 쓰시는 분, Hibernate가 뒤에서 어떤 쿼리를 만드는지 본 적 있으세요?"

실행된 쿼리, 실행 시간, 실행 빈도를 자동 수집. N+1 문제를 숫자로 바로 확인 가능.

스크린샷: WhaTap SQL 통계 — 슬로우 쿼리 Top 10

C. 다음 단계 (2분)

Structured Log가 잘 남아있으면, AI Agent가 활용할 수 있는 기반이 됩니다. Honeycomb MCP, Datadog MCP 등 이미 프로덕션에서 사용 중. 반대로 log.error("결제 실패: " + e.getMessage()) 뿐이면 AI도 할 수 있는 게 없습니다.

"지금 로그 한 줄을 바꾸는 게, 미래에 AI Agent를 활용할 수 있는 기반이 됩니다."

클로징 — 3분

"보이게 만드는 사람"

한 단계씩

  1. Structured Log 어디서, 누구한테 에러가 집중되는지. — log 한 줄만 바꾸면 됨
  2. Tracing 왜 느린지, 어디서 막히는지. — @Observed 한 줄 (Spring Boot)
  3. Dashboard 분포와 추이를 한눈에. — Grafana (무료) or WhaTap
  4. Real-time APM 지금 이 순간 무슨 일이 일어나는지. — WhaTap 같은 APM 도입
