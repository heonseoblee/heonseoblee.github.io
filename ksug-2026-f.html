Part 2

서비스가 커지면 드러나는 것들

28분

서비스 초기에는 다 괜찮았습니다. 회원도 적고, 데이터도 적고, 다 빨랐습니다.
"모니터링? 나중에 하지 뭐."

그런데 서비스가 성장하면서 — 코드는 안 바꿨는데 — 하나씩 문제가 드러납니다.

성장 ① 회원이 늘고, 데이터가 쌓이기 시작하면 APM

슬로우 쿼리 — 원래는 빠른 쿼리였다

회원 수 적고, 회원당 아이템 수가 적을 때는 아무 문제 없었습니다. 그런데 서비스가 계속되면 — 슬로우 쿼리는 다양한 형태로 찾아옵니다.

유형처음엔서비스가 커지면
N+1아이템 10개 → 20ms아이템 1,000개 → 3,200ms
인덱스 미적용컬럼 10개, 인덱스 잘 탐기능 추가로 컬럼 100개, 쿼리 패턴 바뀌면서 인덱스 안 탐
과도한 조인테이블 2~3개 조인, 빠름요구사항 추가되면서 5~6개 조인, 데이터 커지면서 급격히 느려짐
불필요한 중복서비스 레이어 분리, 깔끔각 레이어가 독립적으로 같은 마스터 데이터를 3번씩 조회

공통점은 — 처음부터 느린 게 아니라, 서비스가 성장하면서 느려진다는 겁니다. 코드는 바뀌지 않았는데 쿼리는 점점 느려집니다.

GET /api/v1/orders (3,200ms)
├── findOrder           (5ms)    OK
├── SELECT order        (3ms)    OK
├── SELECT item         (2ms)    OK  ← × 1,000번 반복!
├── SELECT item         (2ms)    OK
│   ...
└── (총 SQL 1,001회)              N+1

이건 N+1의 예시지만, 인덱스 미적용이든 과도한 조인이든 — 트레이스를 열어보면 어떤 쿼리가 얼마나 걸리는지 한눈에 보입니다. 그 전에는 "어디가 느린지" 감으로 찾아야 했습니다.

트레이스(Trace) = 한 요청이 서버 안에서 어디를 거쳐갔는지, 각 구간에 몇 ms가 걸렸는지 따라가는 것.

SQL 통계에서 실행 횟수 내림차순으로 정렬하면 중복 쿼리가 바로 보이고, 평균 수행 시간 내림차순으로 정렬하면 느린 쿼리가 보입니다. 코드를 전부 읽지 않아도 데이터로 먼저 발견할 수 있습니다.

데모: 트레이스 뷰 — 쿼리별 소요시간 / SQL 통계 — 실행 횟수·수행시간 내림차순
코드 리뷰에서는 안 보였다. 트레이스에서 처음 보였다.
쿼리 개선 후 — p95 응답시간 3초 → 300ms.

커넥션 풀 고갈 — 쿼리는 1ms인데 응답이 25초

증상

특정 API가 평소엔 50ms, 가끔 20~30초. 느릴 때 트레이스를 열어보면 — SQL 자체는 1~6ms인데 전체 경과가 25,000ms.

DB Connection 획득 24,931ms
SQL 실행 1~6ms
전체 경과 25,057ms

거의 전부가 커넥션 대기 시간. 쿼리 자체는 정상.

커넥션 풀(Connection Pool) = DB에 연결할 수 있는 통로 개수. 예를 들어 20개. 다 사용 중이면 새 요청은 빈 통로가 생길 때까지 줄을 서야 합니다.

왜 갑자기?

초기: 풀 20개, 동시 3~5개 → 문제 없음
사용자 증가: 동시 18~20개 → 대기 25초, 타임아웃

Spring Boot의 HikariCP 기본 maximumPoolSize=10. 서비스가 커지면 부족해집니다.

메트릭스 차트에서 DB Pool Count와 DB Active Count를 나란히 보면 — Active가 Pool 한도에 딱 붙어있는 게 보입니다.

데모: 와탭 트레이스 — DBC 대기 24,931ms / DB Pool vs Active 차트
쿼리 튜닝만이 답이 아니다.
코드 바깥 — 인프라 설정 문제를 트레이스에서 발견했다.
로그에도, 슬로우 쿼리 로그에도 안 잡힌다.

힙 메모리 부족 — GC가 CPU를 잡아먹고 있었다

증상

서비스가 전체적으로 느려짐. 특정 API가 아니라 모든 요청이 느림. CPU는 100%인데, 트레이스에서 느린 구간이 안 보임. 에러도 없음.

이런 경우 서버 메트릭을 봐야 합니다.

힙 메모리(Heap Memory) = JVM이 객체를 저장하는 공간. -Xmx 옵션으로 최대 크기를 정합니다. 이 공간이 가득 차면 JVM이 안 쓰는 객체를 정리하는데, 이걸 GC(Garbage Collection)라고 합니다. GC가 돌 때 CPU를 씁니다.

서버 모니터링에서 보이는 것

힙 메모리 사용률 92% (거의 가득)
GC 횟수 분당 200회 (평소 5회)
CPU 사용률 98% (GC 때문)

힙 메모리 그래프가 톱니 모양인데 바닥이 점점 올라가는 패턴 — GC를 돌려도 메모리가 회수되지 않고 있다는 뜻입니다. 이게 계속되면 결국 OutOfMemoryError로 서비스가 죽습니다.

커넥션 풀과 같은 구조

초기: -Xmx512m, 충분 → 문제 없음
데이터/트래픽 증가 → 힙 사용량 점점 증가 → GC 폭풍 → CPU 100% → 전체 느려짐

커넥션 풀과 똑같습니다. 코드는 정상인데, 설정이 서비스 규모를 못 따라간 것.

해결도 비슷합니다:

  • 당장 급한 조치 — -Xmx 늘리기 (서버 설정 변경)
  • 근본 원인 — 메모리 누수나 과도한 객체 생성이 있는지 확인 (힙 덤프 분석)
데모: 와탭 힙 메모리 그래프 — 톱니 바닥이 올라가는 패턴 / GC 횟수 급증
커넥션 풀 부족, 힙 메모리 부족 — 둘 다 코드가 아니라 서버 설정 문제입니다.
APM 트레이스만으로는 안 보이고, 서버 메트릭을 봐야 원인을 찾을 수 있습니다.

성장 ② 서비스가 복잡해지면 APMLOG

외부 연동이 늘어난다

서비스가 커지면 결제 API, 택배 API, 알림 서비스 등 외부 연동이 늘어납니다.

내부 로직 + SQL 120ms
외부 결제 API 4,800ms

트레이스에서 HTTP Call 요약을 보면 — 외부 결제 API 1건이 합계 시간의 90%.

중요한 건 "우리 잘못이 아니라는 걸 입증하는 것"입니다. APM 트레이스 한 장이면 끝납니다.

데모: 트레이스 HTTP Call 요약 — 외부 API 지연 비중

배포 배포는 빈번해지고, 개발팀도 늘어나면 APM

배포 전후 비교 — 개선됐는지 어떻게 아나

리팩토링 완료 → "아마 빨라졌겠지" → PR 머지. 이러고 계시진 않나요?

배포 전 (v2.0)

p50: 180ms
p95: 3,200ms
DB calls/req: 8.3

배포 후 (v2.1)

p50: 160ms
p95: 290ms
DB calls/req: 1.1

여기까지는 좋았는데 —

[DEPLOY v2.1.1] ─────────────────────────
p50: 170ms  p95: 8,400ms  ← 신규 기능 배포 후 급등

개발팀이 늘어나면 서로 다른 팀이 같은 서비스에 배포합니다. 예상 못한 regression이 배포 직후에 잡힙니다.

p95 = 100명의 요청을 빠른 순서로 세웠을 때 95번째 사람의 응답시간. "상위 5%의 최악 경험"을 보는 지표. 평균이 아니라 이걸 봐야 진짜 문제가 보입니다.

"어제 배포 때문이야" → 아닌데?

장애 슬랙 알람 → "어제 배포가 원인이다" 주장 → 에러 통계에서 최초 생성 시간 확인 → 해당 에러는 배포 14일 전부터 있었음.

배포 후 캐시 초기화로 나오는 캐시 미스를 에러로 로깅하고 있었다면 — 평소에는 넘어가던 것들이, 장애 난 날에는 범인처럼 보입니다.

데모: 에러 통계 — 최초 생성 시간 / 배포 전후 응답시간 그래프
에러는 발견된 날이 발생한 날이 아닐 수 있습니다.
배포가 "끝"이 아니라 "확인의 시작"이 되면 — 이런 혼란이 줄어듭니다.

  • 사장님 PC에서만 느린 경우 — 평균은 괜찮다고 했는데 큰일
  • 헤비 유저가 느린 경우 — 돈 되는 고객이 이탈
  • 영업이 컴플레인을 가져오기 전에 먼저 발견하면 좋겠죠

분포에서 "긴 꼬리" 구간 드릴다운 → 특정 조건에서만 느린 것을 발견.

비정상 트래픽 — 기계가 API를 폭격하고 있었다

슬로우 쿼리 없음, 개별 트랜잭션 정상, 근데 DB 부하가 높음. IP 트랜잭션 통계에서 호출 건수 내림차순 → 내부 IP 한 개가 전체 요청의 60%. 고객 부하 테스트, 배치 잡 루프 버그 — 다양합니다.

데모: 응답시간 히스토그램 P50/P95/P99 / IP 트랜잭션 통계
평균은 거짓말한다. 분포가 진실을 말한다.
보는 방법이 달라지면, 질문하는 방식이 달라진다.

패턴 다양한 서비스, 다양한 패턴 — 히트맵 APM

"그래프 모양 자체가 진단이다"

히트맵 = 요청 하나하나를 점으로 찍어서, 가로축은 시간, 세로축은 응답시간으로 나열한 그림. 점이 많이 모인 곳이 밝게 빛납니다.
패턴모양의미
세로줄│ 특정 시각에 집중Lock 해소 → 대기 트랜잭션 한꺼번에 종료
가로줄── 특정 응답시간에 집중타임아웃 설정값이 드러남 (예: 10초)
폭주 패턴▲ 전체적 상승과부하 → 응답시간 전체 증가
플라잉 패턴〜 파도처럼 반복공유 자원 간헐적 부족
데모: 히트맵 패턴 4종 예시

세로줄 분기점

단일 인스턴스에서만? → AP 내부 Lock. 모든 인스턴스에서 동시? → 외부 DB Lock. 이것만으로 조사 범위가 반으로.

CPU 100% — 문제도 안 나오고, 트레이스도 안 나오는데

원인확인신호
힙 메모리 가득 참힙 메모리 그래프GC 반복 → CPU 급증
실제 요청이 많음TPS 지표요청 수와 CPU 비례
코드 내 비효율스레드 덤프무한 루프, 과도한 직렬화

스레드 덤프 → CPU 시간 내림차순 정렬 → CPU 많이 쓰는 스레드 특정 → 자주 호출되는 메소드 확인.

데모: 스레드 덤프 — CPU 시간 내림차순 / 힙 메모리 GC 패턴
"왜 느리지?"가 아니라 "이건 세로줄인가, 폭주인가"로 시작하게 됩니다.
데이터를 읽는 방법을 알면, 질문하는 방식이 달라집니다.

APM은 첫 번째 바만 봅니다. 나머지는?

  • 네트워크 / CDN — 특정 리전에서만 느림
  • 중간 단계 — 로드밸런서, 프록시, WAF 등에서의 지연
  • JS 번들 — 모바일에서 치명적
  • 클라이언트 코드 — 프론트엔드 렌더링 병목, 서드파티 스크립트
RUM (Real User Monitoring) = 실제 사용자의 브라우저에서 성능을 측정하는 것. 서버가 "나는 빨리 보냈어"와 사용자가 "화면이 떴다"는 다릅니다.
데모: 브라우저 모니터링 — 페이지 로드 타임 분해 (서버 / 네트워크 / 렌더링)
APM과 DB로 서버 안의 문제는 찾을 수 있습니다.
하지만 서버와 사용자 사이 — 네트워크, CDN, 브라우저, 클라이언트 코드 —
이건 RUM이 아니면 안 보입니다.

AI가 분석해줘도, 읽을 줄 알아야 한다

AI가 이렇게 말했을 때:

"DB Connection Pool의 Active Count가 maximumPoolSize에 도달하여
커넥션 획득 대기가 발생하고 있습니다."

이걸 이해하려면 — 오늘 이야기한 정도는 알아야 합니다.
AI가 읽어줘도, 내가 모르면 활용할 수 없습니다.

"나는 모니터링을 해야 되겠구나"라고 생각이 드는 것.
그게 오늘 발표의 목표입니다.

데모 — 와탭으로 배포 후 1시간 동안 볼 수 있는 것들

실제로 배포하고 나서, 1시간 동안 와탭에서 뭘 볼 수 있는지 살펴보겠습니다.

① 히트맵 — 배포 직후 응답시간 변화

배포 마커 기준으로 전/후 히트맵 패턴 비교. 점들이 위로 올라갔는지, 세로줄이 생겼는지.

② 트레이스 — 느린 요청 하나 열어보기

히트맵에서 느린 점 클릭 → 트레이스 열기 → SQL이 느린지, 커넥션 대기인지, 외부 API인지 바로 확인.

③ SQL 통계 — 실행 횟수, 수행 시간 정렬

배포 전에 없던 쿼리가 상위에 올라왔는지, 기존 쿼리의 수행 시간이 늘었는지.

④ 메트릭스 — DB Pool, 에러율, TPS

커넥션 풀이 포화되고 있는지, 에러율이 올라갔는지, TPS 변화는 어떤지 한 화면에서 확인.

⑤ 에러 통계 — 새로 생긴 에러 확인

최초 생성 시간 기준으로 정렬 → 이번 배포로 새로 생긴 에러만 필터링.

라이브 데모: 와탭 APM — 배포 후 1시간 체크 루틴
오늘 본 사례들 — 전부 이 화면들 안에 있습니다.
트레이스를 한번 보면, 다시 안 보던 시절로 돌아갈 수 없습니다.

📋 한 장 요약

서비스가 커지면드러나는 문제찾는 방법
데이터가 쌓이면원래 빠르던 쿼리가 느려짐트레이스 + SQL 통계
사용자가 늘면커넥션 풀 부족, 대기 25초트레이스 + 메트릭스
트래픽이 늘면GC 폭풍 → CPU 100% → 전체 느림힙 메모리 + GC 메트릭
연동이 늘면외부 API 지연, 재현 불가 버그HTTP Call 추적 + 구조화 로그
팀이 늘면배포 후 regression, 에러 공방배포 전후 비교 + 에러 통계
고객이 늘면대용량 고객만 느림, 비정상 트래픽분포 분석 + IP 통계
다양한 서비스서비스별 다른 장애 패턴히트맵 패턴 분석
위 전부 정상인데고객이 느리다고 함RUM
